# Monday_parse — Документация проекта

## Что делает этот инструмент

**Monday_parse** — это серверный API на Node.js/Next.js, который принимает PDF-файл страхового полиса в формате base64, извлекает из него ключевые данные и возвращает структурированный JSON.

### Входные данные
HTTP POST-запрос с телом:
```json
{ "pdfData": "<base64-строка PDF>" }
```

### Выходные данные
```json
{
  "success": true,
  "result": "1223|2814919278|281491927",
  "details": {
    "insurance_type": "Зелена картка",
    "price": "1223",
    "ipn": "2814919278",
    "policy_number": "281491927",
    "insured_name": "KARAINAKI OLEKSANDR",
    "start_date": "04.06.2026, 00:00",
    "end_date": "18.06.2026",
    "car_model": "VOLKSWAGEN TOURAN",
    "car_number": "BH0720TH",
    "vin_number": "WVGZZZ1TZAW084872",
    "phone": "380963694232"
  }
}
```

Поле `result` — строка `price|ipn|policy_number` для быстрой вставки в Monday.com.

## Поддерживаемые типы полисов

### 1. Автоцивілка (ОСЦПВ)
- Стандартний внутрішній поліс цивільно-правової відповідальності
- Формати номерів: `EP-XXXXXXXXX`, `611933-XXXX-XXXXXXXXX` (9 цифр в кінці)
- Дата починається з часу: `З 00:00 04 січня 2025 р.`
- Ціна: секція `15 Розмір страхової премії` або `Страховий платіж`
- Ім'я: блок `3. СТРАХУВАЛЬНИК` або `Найменування`

### 2. Зелена картка (Green Card)
- Міжнародне страхування, діє за кордоном
- Формати номерів: `UA/078/41812487`, `UA048.41846155`, `Поліс UA/ 42320944`
- Ім'я може бути латиницею
- Підтримуються формати різних страховиків:
  - Стандартний: секція `найменування Страхувальника`
  - Картковий: секція `ПІБ ТА АДРЕСА СТРАХУВАЛЬНИКА`
  - Княжа: секція `3СТРАХУВАЛЬНИК` → ім'я латиницею на наступному рядку

### 3. Повний Автозахист
- Комплексний договір страхування, надбудова над ОСЦПВ-полісом
- Номер формату: `FO-XXXXXXXX` (наприклад, `FO-03394998`)
- Страхувальник: не зазначений у тілі документу — лише скорочене ФІО у рядку підпису (`ДУДНІК О. А.`)
- Телефон: у секції `Підписано за допомогою одноразового ідентифікатору ... відправленого на номер +380...`
- Ціна: рядок `«Автоцивілка плюс»50 000,00200,00` — другe число (премія)
- Посилається на базовий ОСЦПВ-поліс

### 4. Travel (Мандрівник)
- Міжнародний договір комплексного страхування під час перебування за кордоном
- Номер формату: `611933-4421-XXXXXXX` (наприклад, `611933-4421-1000090`)
- Страховик: Євроінс Україна
- Ім'я: з таблиці `П.І.Б. / Name` (латиниця або кирилиця)
- Телефон: поле `Тел.:`
- Ціна: поле `13. Загальний страховий платіж, грн.`
- Немає авто (car_model, car_number, vin → null)

// Пріоритет: Повний Автозахист → Зелена картка → Travel → Автоцивілка

### 2026-06-04 — Fix Зелена картка Княжа (коміт `bca1f63`)

### 2026-06-04 — Fix car_model порожній для Євроінс-формату (коміт `02d2b68`)
- **Проблема**: `car_model: ""` для полісу `No233812122` (Євроінс Україна)
- **Причина**: у PDF Євроінс поля `9.2. Марка` і `9.3. Модель` — об'єднана комірка (merged cell).
  `pdf-parse` розміщує `Volkswagen Touran` рядком після `9.2. Марка`, а `9.3. Модель` залишається порожнім.
  Старий паттерн Format 2 вимагав оба поля заповненими → не знаходив.
  Format 3 знаходив мусор (`"  "` + `" "`) → після trim давав `""`.
- **Рішення**: додано паттерн **Format 2b** між Format 2 і Format 3:
  ```javascript
  // Бренд+модель на наступному рядку після "9.2. Марка", поле 9.3 порожнє
  const m = fullText.match(/9\.2\.\s*Марка[^\n]*\n([^\n]+)/i);
  if (m) { const val = m[1].trim(); if (val && !/^9\.\d/.test(val)) carModel = val; }
  ```
- Протестовано на: `No233812122`, `EP-238523789` (Audi A6), `611933-2212` (Nissan Maxima), Зелена картка

### 2026-07-05 — Fix Зелена картка: ціна для формату ПЗУ
- **Проблема**: `price` повертав неправильне значення для полісу `42513018.pdf` (ПЗУ). Повертав `3720` замість правильних `16022`.
- **Причина**: У цьому форматі PDF ціна `16022.00` знаходиться на ~2300 символів нижче за заголовок `Розмір страхової премії` (втрачається структура таблиці при транскрибації). Старий regex захоплював числа з попередніх секцій (наприклад, `3720` з дати `05.07.2026`).
- **Рішення**: Замінено три regex-патерни на two-step підхід:
  1. Знаходимо позицію `Розмір страхової премії` в тексті
  2. Беремо блок 3000 символів після заголовку
  3. Шукаємо число з `(?=[,.]00)` — патерн `(\d[ \t\d]*\d)(?=[,.]00)`
  4. Fallback: якщо `.00` не знайдено, шукаємо число на окремому рядку
- **Важливо**: `[ \t\d]*` замість `[\d\s]*` — щоб не захоплювати цифри з інших рядків (переноси рядків)
- Протестовано на: `30105159` (2510), `YFDe4d7vMq` (12775), `policy` (1926), `BqpKUV7jrA` (1199), `42513018` (16022), `41846155.txt` (11442)
- Інші типи полісів (Автоцивілка, Повний Автозахист, Travel) не змінювались і працюють коректно

### 2026-07-05 — Fix Зелена картка: start_date та end_date
- **Проблема**: `start_date` повертав неправильну дату (брав з "Дата внесення запису" замість секції 5.1) або `null`. `end_date` міг хибно співпасти.
- **Причина**: Поточні regex шукали дати від `5.1` до ПЕРШОЇ комбінації час+дата, але вона могла бути з іншого блоку. Для форматів `Договоруз25.10.2025` (без пробілу) GC-патерн вимагав `з\s+` (з + пробіл).
- **Рішення**: Додано два нових патерни для Green Card:
  - start_date: `Договоруз[ \t\n]*DD.MM.YYYY` та `Договору з DD.MM.YYYY`
  - end_date: `Договорупо[ \t\n]*DD.MM.YYYY` та `Договору по DD.MM.YYYY`
  - Патерни працюють для форматів з пробілом і без нього, для дат на одному рядку та на наступному
- Протестовано на: `30105159` (25.10.2025/24.03.2026), `YFDe4d7vMq` (21.05.2026/20.05.2027), `policy` (03.02.2026/02.03.2026), `BqpKUV7jrA` (15.05.2026/29.05.2026), `42513018` (11.07.2026/10.07.2027), `41846155.txt` (12.02.2026/11.02.2027)
- Інші типи полісів не змінювались

## Файл парсера

`api/extract.js` — єдиний файл з усією логікою. Структура: inline if-else блоки з каскадними `||` для кожного поля. Нові типи додаються через окремі `if (!field && isXxx)` блоки в кінці кожної секції — стара логіка не зачіпається.
